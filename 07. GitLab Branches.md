## GitLab Branches

- It is a separate line of development in a Git repository.
- It allows developers to work on new features or bug fixes without affecting the main code.

---

### Branch Workflow

```text
main
 │
 ├───────────────► feature-login
 │
 ├───────────────► feature-payment
 │
 └───────────────► bugfix-header
```

Each developer works on a separate branch.

---

### Default Branch

When a project is created, GitLab creates a default branch.

Usually, it is:

```text
main
```

The `main` branch contains stable and production-ready code.

---

### Create a Branch

Using Git:

```bash
git switch -c feature-login
```

Push the branch:

```bash
git push origin feature-login
```

---

### View Branches

Local branches:

```bash
git branch
```

Remote branches:

```bash
git branch -r
```

All branches:

```bash
git branch -a
```

---

### Switch Branch

```bash
git switch feature-login
```

Or:

```bash
git checkout feature-login
```

---

### Delete a Branch

Delete a local branch:

```bash
git branch -d feature-login
```

Delete a remote branch:

```bash
git push origin --delete feature-login
```

---

### Merge a Branch

After completing the work:

1. Push the branch.
2. Create a Merge Request.
3. Review the code.
4. Merge into the `main` branch.

---

### Branch Naming Convention

Use meaningful names.

Examples:

```text
feature-login
feature-payment
bugfix-header
bugfix-api
hotfix-security
release-v1.0
```

---

### Example

A team is developing an online shopping application.

- Alice creates `feature-login`
- Bob creates `feature-cart`
- Charlie creates `bugfix-payment`

Each developer works independently.

After testing, each branch is merged into the `main` branch through a Merge Request.

---

### Key Points

- A branch is an independent line of development.
- The `main` branch contains stable code.
- Developers create feature branches to work safely.
- After review, branches are merged into the `main` branch.
- Use meaningful branch names and delete merged branches to keep the repository clean.
